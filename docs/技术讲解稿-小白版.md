# 监狱智能行为分析系统 — 技术讲解稿（小白版）

## 一句话概括

> 用摄像头实时拍画面，AI 自动识别异常行为（打架、跌倒、离岗），发现问题立刻报警。

---

## 整体架构（像流水线一样）

```
摄像头 → 视频流中转站(go2rtc) → 大脑(Java后端) → 眼睛(AI检测) → 屏幕(前端页面)
                                                    ↑
                                                  数据库(记住所有摄像头信息)
```

就像一条工厂流水线，每个环节各干各的活。

---

## 5 个核心模块，逐个拆解

### 1. 前端页面 — "屏幕"（你看得见的部分）

**用什么做的：** React + Tailwind CSS

**通俗解释：**
- React 是 Facebook 团队做的一个"搭积木"工具，把网页拆成一块块小组件
- Tailwind CSS 是"样式速写笔"，不用单独写 CSS 文件，直接在代码里写 `bg-red-500` 就是红色背景
- Vite 是"打包机"，把写的代码压缩成浏览器能读的文件

**比喻：** 前端就像汽车的仪表盘，你只看指针和数字，不需要知道发动机怎么转的。

**它干了什么活：**
- 监控大屏：显示所有摄像头的实时画面
- 设备管理：添加/删除摄像头
- 报警中心：显示 AI 检测到的异常
- 数据分析：用图表展示统计趋势

---

### 2. Java 后端 — "大脑"（指挥中心）

**用什么做的：** Spring Boot（Java 框架）

**通俗解释：**
- Java 是一门老牌编程语言，稳得像银行系统
- Spring Boot 是 Java 的"脚手架"，帮你把 80% 的重复工作都做好了，你只管写业务逻辑
- 它跑在 5000 端口上，像一个 24 小时营业的客服中心

**比喻：** 后端就像餐厅前台，客人（前端）点菜，前台通知厨房（Python AI）做菜，做好了端上来。

**它干了什么活：**
- 提供 API 接口：前端问"有哪些摄像头？"，后端去数据库查完返回
- 管理摄像头：增删改查，存在数据库里
- 管理视频流：告诉 go2rtc 哪些摄像头需要转发视频
- 用户认证：登录、JWT 令牌（类似门票，有效期内不用重复登录）
- 转发检测结果：Python 检测到异常 → 后端 → 前端弹出报警

---

### 3. H2 数据库 — "记事本"（记住摄像头信息）

**用什么做的：** H2 Database（嵌入式数据库）

**通俗解释：**
- H2 是一个"口袋数据库"，不需要单独安装 MySQL 那样的数据库服务
- 它直接跑在 Java 程序里，数据存成一个文件（就像 Excel 表格）
- 以前用 JSON 文件记摄像头，现在升级成数据库了，查得快、不怕多人同时改

**比喻：** 以前用笔记本手写记账（JSON 文件），现在换成 Excel（数据库），查找修改都方便多了。

**它存了什么：**

| 字段 | 含义 | 举例 |
|------|------|------|
| id | 编号 | cam0, hik-01 |
| name | 名字 | "大门摄像头" |
| type | 类型 | usb / rtsp |
| ip | IP 地址 | 192.168.1.64 |
| status | 状态 | online / offline |
| go2rtc_id | 视频流编号 | cam_hik01 |

---

### 4. go2rtc — "视频中转站"

**用什么做的：** go2rtc（开源流媒体工具，Go 语言写的独立程序）

**通俗解释：**
- 摄像头的视频流（RTSP 协议）不能直接在浏览器播放，就像 DVD 不能直接在手机上看
- go2rtc 就是个"转码中转站"，把摄像头的 RTSP 流转成浏览器能播的 WebRTC 流
- 它是独立的程序（go2rtc.exe），Java 启动时顺手把它也拉起来

**比喻：** 摄像头说英文（RTSP），浏览器只懂中文（WebRTC），go2rtc 就是翻译官。

**三个端口各管各的：**

| 端口 | 作用 | 比喻 |
|------|------|------|
| 1984 | 管理接口 | 前台，告诉它加哪个摄像头 |
| 8554 | RTSP 输出 | 后门，Python AI 从这里拿视频做检测 |
| 8555 | WebRTC 输出 | 正门，浏览器从这里看直播 |

**为什么需要它？**
- 以前：Python 直接连摄像头 → 一路摄像头占一路带宽，10 路就崩了
- 现在：摄像头只连 go2rtc 一份，go2rtc 再分发给 Python 和浏览器 → 省带宽、延迟低

---

### 5. Python AI 检测 — "眼睛"（识别异常行为）

**用什么做的：** YOLOv8 + OpenCV + PyTorch

**通俗解释：**
- **YOLOv8** 是一个"看图识物"的 AI 模型，专门识别人体姿态（站、坐、躺、打架等）
- **OpenCV** 是"摄像机驱动"，负责从视频流里一帧一帧抓画面
- **PyTorch** 是"AI 引擎"，让 YOLO 跑在显卡（GPU）上，速度快 10 倍

**比喻：** YOLOv8 是训练有素的保安，盯着监控画面看，发现异常就举手报告。

**它能检测 6 种异常：**

| 异常行为 | 怎么判断的 |
|----------|-----------|
| 跌倒 | 人体骨架倾斜角度超过阈值 |
| 打架 | 短时间内多人快速移动 + 肢体碰撞 |
| 疲劳 | 头部持续低垂超过设定时间 |
| 眼疲劳 | 眼睛纵横比 (EAR) 低于阈值 |
| 离岗 | 指定区域长时间没有人 |
| 聚集 | 一个画面里人太多了 |

**工作流程：**
1. 从 go2rtc 拉视频流（`rtsp://localhost:8554/cam_01`）
2. 每 2 帧做一次检测（不用每帧都检测，省 GPU）
3. 识别人体骨架（17 个关键点：头、肩、肘、腕、膝、踝等）
4. 根据骨架姿态判断有没有异常
5. 有异常 → 把结果（JSON）+ 截图（JPG）发给 Java 后端
6. 后端推送到前端 → 弹出报警

---

## 关键技术名词解释

| 名词 | 全称 | 小白翻译 |
|------|------|----------|
| RTSP | Real Time Streaming Protocol | 摄像头的"视频通话协议"，就像微信视频 |
| WebRTC | Web Real-Time Communication | 浏览器的"视频通话协议"，延迟超低（<500ms） |
| API | Application Programming Interface | 不同程序之间的"传话筒" |
| JWT | JSON Web Token | 登录后的"门票"，一段时间内不用重复登录 |
| MJPEG | Motion JPEG | 一串 JPEG 图片快速切换，像翻页动画 |
| ONVIF | Open Network Video Interface | 摄像头界的"普通话"，统一标准让不同品牌互通 |
| SSE | Server-Sent Events | 服务器主动给浏览器发消息，像手机推送通知 |
| GPU | Graphics Processing Unit | 显卡，AI 计算比 CPU 快 10-50 倍 |
| CUDA | Compute Unified Device Architecture | NVIDIA 显卡的"AI 加速开关" |

## 数据流向（一个报警的完整旅程）

```
1. 摄像头拍到有人打架
       ↓
2. go2rtc 收到 RTSP 视频流
       ↓
3. Python 从 go2rtc 拉帧，YOLO 识别出"打架"
       ↓
4. Python 把结果 POST 到 Java 后端（/api/update_frame）
       ↓
5. Java 后端存到文件，通过 SSE 推送给前端
       ↓
6. 前端收到推送，页面弹出红色报警 + 截图
       ↓
7. 保安看到报警，去现场处理
```

整个过程不到 2 秒。

## 为什么要这样设计？（技术选型的理由）

| 选择 | 原因 |
|------|------|
| Java 后端（不是 Node.js） | 稳定、企业级、团队熟悉 |
| React 前端（不是 Vue） | 组件化强、生态丰富 |
| H2 数据库（不是 MySQL） | 零配置、内嵌运行、监狱环境无需额外装数据库服务 |
| go2rtc（不是直接拉流） | 统一管理、支持 WebRTC 低延迟、减轻摄像头压力 |
| YOLOv8（不是传统算法） | 准确率高、速度快、支持 GPU 加速 |
| 三组件分离 | 各自独立运行、互不影响、坏了哪个修哪个 |
