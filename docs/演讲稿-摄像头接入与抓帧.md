## 1. 接进来 — cameras.json + 三种接入方式（2min）

大家好，今天我给大家讲一下我们这个系统是怎么把摄像头接进来的。

整个过程就四个字：**接进来、读出来、传上去、展出来**。

### 怎么接？

我们系统支持三种摄像头接入方式：

| 方式 | 场景 | 地址格式 |
|------|------|---------|
| **RTSP** | 网络摄像头（海康/大华），最常用 | `rtsp://192.168.1.64:554/stream` |
| **USB** | 本地 USB 摄像头，演示用 | 设备号 `0`、`1` |
| **HTTP** | 网络快照摄像头 | `http://192.168.1.64/snapshot.jpg` |

监狱场景主要用 **RTSP**——摄像头插网线，给一个 IP 地址，用 RTSP 协议传视频流。就像给摄像头打视频电话："把你拍到的画面发给我"。

### 配置在哪？

所有摄像头信息存在 `detection/cameras.json` 里，Python 和 Java 共享同一份配置：

```json
[
  {
    "id": "cam_01",
    "name": "大门摄像头",
    "type": "rtsp",
    "rtsp_url": "rtsp://admin:pass@192.168.1.64:554/Streaming/Channels/101",
    "location": "大门"
  }
]
```

Python 启动时读这个文件，知道要连哪些摄像头。
Java 后端也读这个文件，给前端展示摄像头列表。

**代码位置：**
- Python 读配置：`detection/yolov8_security.py:270`，`load_cameras_config()` 方法
- Java 管理配置：`server/.../CameraConfigController.java:28`，增删改查接口

## 2. 读出来 — cv2.VideoCapture + 跳帧推理（2min）

摄像头连上了，Python 程序要开始读画面了。

### 用什么读？

Python 用一个叫 **OpenCV** 的库，提供了一个特别简单的接口：

```python
cap = cv2.VideoCapture(source)  # source 是 RTSP 地址或 USB 设备号
ret, frame = cap.read()         # 读一帧画面
```

就这两行。`cap.read()` 每次返回一帧画面，就像翻页动画书，翻得够快（每秒 25-30 页）就是流畅的视频。

### 关键问题：每帧都检测吗？

不是。每帧都做 AI 检测，显卡扛不住。

我们用**跳帧推理**——每 2 帧检测 1 次，中间那帧直接复用上一次的结果。

打个比方：你看翻页动画书，第 1 页仔细看了，第 2 页和第 1 页几乎一样，直接用第 1 页的结论就行。

**效果：** 推理开销直接减半。

```python
INFERENCE_EVERY = 2  # 每2帧推理1次

while True:
    ret, frame = cap.read()
    frame_count += 1

    if frame_count % INFERENCE_EVERY == 1:
        results = model(frame)       # 真正做 AI 推理
        cached_results = results     # 缓存
    else:
        results = cached_results     # 复用上次结果，省一半 GPU
```

**代码位置：**
- 打开摄像头：`detection/yolov8_security.py:1343`，`cap = cv2.VideoCapture(source)`
- 跳帧推理：`detection/yolov8_security.py:108`，`INFERENCE_EVERY = 2`
- 主循环：`detection/yolov8_security.py:1520`

## 3. 传上去 — HTTP POST JPEG → Java 接收 → MJPEG 流输出（2min）

帧读出来了，检测也做完了。但 Python 跑在后台，用户看不到。用户看到的是浏览器里的网页。

所以要把画面**传上去**。

### Python → Java 后端

Python 把每一帧压缩成 JPEG 图片，通过 HTTP POST 发给 Java 后端：

```python
# detection/yolov8_security.py:1736
frame_resized = cv2.resize(frame, (960, 540))     # 缩小到 960x540，省流量
_, img_encoded = cv2.imencode('.jpg', frame_resized,
                              [cv2.IMWRITE_JPEG_QUALITY, 50])  # JPEG 质量 50%
requests.post(
    "http://localhost:5000/api/update_frame",
    files={'frame': ('frame.jpg', img_encoded.tobytes(), 'image/jpeg')},
    timeout=0.5    # 500ms 超时，不卡主循环
)
```

> 960x540 + quality=50，一帧大概 30-50KB，每秒传 15 帧，带宽不到 1MB/s。

### Java 后端 → MJPEG 流

Java 后端收到帧后，缓存在内存里，然后通过 MJPEG 流输出。

MJPEG 就是**一串 JPEG 图片快速切换**，像翻页动画。前端用一个 `<img>` 标签就能播放：

```tsx
<img src="/video_feed?cam=0" />
```

**代码位置：**
- Python 发送帧：`detection/yolov8_security.py:1736`
- Java 接收帧：`server/.../StatsController.java:301`，`/api/update_frame` 接口
- Java MJPEG 输出：`server/.../VideoStreamController.java:100`，`/video_feed` 接口

## 4. 展出来 — 前端 img 标签拉 MJPEG 流 + go2rtc WebRTC 备选（1min）

### MJPEG 方式（主方案）

最简单的方式——一个 `<img>` 标签：

```tsx
<img src="/video_feed?cam=0" className="w-full h-full object-cover" />
```

浏览器原生支持 MJPEG，不需要任何插件。延迟大约 1-2 秒。

### WebRTC 备选方案（go2rtc）

如果需要更低延迟，可以用 **go2rtc** 做 WebRTC 播放，延迟降到 **不到 0.5 秒**。

go2rtc 是一个独立的流媒体中转站：
- 摄像头只连 go2rtc 一次
- go2rtc 分发给 Python（RTSP）和浏览器（WebRTC）
- 自动重连，摄像头掉线不会卡死

**代码位置：**
- 前端 MJPEG：`web/src/pages/Monitor.tsx:422`
- go2rtc 管理：`server/.../Go2rtcService.java:119`

## 5. 数据流总结（1min）

用一张图总结整个数据流：

```
摄像头（海康/大华/USB）
    │
    │  RTSP 视频流
    │
    ▼
Python 抓帧 (cv2.VideoCapture)
    │
    ├──→ resize 到 960x540
    │    JPEG 压缩 (quality=50)
    │         │
    │         │ HTTP POST
    │         ▼
    │    Java 后端 (/api/update_frame)
    │         │
    │         │ 缓存帧 → MJPEG 流
    │         ▼
    └──→ /video_feed MJPEG 输出
              │
              ▼
         前端 <img> 标签显示
```

```
摄像头画面
    │
    ▼
每 2 帧取 1 帧 → YOLOv8 推理
    │
    ├── 检测到异常 → 存 JSON + 截图 → Java 后端 → SSE 推送 → 前端报警
    │
    └── 正常 → 只传帧到后端 → MJPEG 流 → 前端显示
```

整个过程：
1. **接进来**：摄像头插网线，RTSP 协议传视频，cameras.json 统一配置
2. **读出来**：Python 用 `cap.read()` 一帧一帧读，每 2 帧推理一次
3. **传上去**：压缩成 JPEG，HTTP 发给后端
4. **展出来**：后端输出 MJPEG 流，前端 `<img>` 标签显示

从摄像头拍到画面，到用户在网页上看到，整个过程不到 **2 秒**。
